# 校园智能服务多 Agent 助手系统概览

## 系统定位

校园智能服务多 Agent 助手系统面向全校师生，提供政策咨询、校园事务办理、预约申请、办理记录查询、反馈投诉和服务建议等能力。

系统不是单一问答机器人，而是由 Supervisor Agent 统一分析用户意图，并把任务分发给不同子 Agent：

- Consult Agent：负责政策、流程、通知公告、校园资源和常见问题咨询。
- Order Agent：负责事务办理、预约申请、办理记录查询、备注修改和取消。
- Feedback Agent：负责投诉、差评、建议、评价和情绪安抚。

## 典型场景

### 政策咨询

用户可以询问：
- 奖学金如何申请？
- 困难生认定需要哪些材料？
- 转专业流程是什么？
- 图书馆研讨间预约规则是什么？

咨询类问题会优先检索校园知识库，并结合用户长期偏好组织回答。

### 事务办理

用户可以发起：
- 预约图书馆研讨间。
- 预约心理咨询。
- 办理在读证明。
- 补办校园卡。
- 查询或取消自己的办理记录。

事务办理通过 MCP Server 暴露工具，由 Service 层完成校验，再通过 MyBatis 操作 MySQL 中的办理记录。

### 反馈投诉

用户可以提交：
- 对图书馆服务的意见。
- 对宿舍维修慢的投诉。
- 对教务服务流程的建议。
- 对校园卡补办体验的评价。

反馈类问题会记录到反馈表中，并可补充处理方案。

## 设计原则

### 职责拆分

不同校园问题类型差异明显。政策咨询、事务办理、投诉反馈所需工具和回答方式不同，因此系统采用 Supervisor Agent + 多子 Agent 的架构，降低单个 Agent 的提示词复杂度和工具混乱风险。

### MCP 工具封装

Agent 不直接操作 Service 或数据库，而是通过 MCP Server 调用业务工具。这样可以让 Agent 专注于理解和决策，Service 层负责参数校验、权限校验和业务规则，数据库负责结构化数据持久化。

### 长期记忆

Memory MCP Server 用于保存用户长期偏好，例如用户喜欢流程图式回答、常在下午预约服务、经常关注奖学金政策等。长期记忆用于提升个性化体验，但不能替代业务校验。

### 知识库增强

校园政策和办事流程会发生变化，咨询 Agent 通过知识库检索降低模型胡编风险。回答应基于知识库和工具结果，信息不足时应提示用户联系相关部门确认。

## 注意事项

- Prompt 用于约束 Agent 行为，但不是安全边界。
- 真实权限控制应放在 Service 层、Mapper 层或后续认证鉴权模块中。
- 对取消记录、修改备注等操作，应先确认 userId 和记录编号。
- 对情绪强烈或投诉类问题，应先安抚，再记录反馈，避免直接承诺不可控的处理结果。
